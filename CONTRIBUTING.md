# Contributing to attesta-protocol

Thanks for your interest in Attesta. This document covers the contract layer
(this repository); the `circuits/` directory carries a stricter
[CONTRIBUTING addendum](./circuits/CONTRIBUTING.md) of its own.

## The standing invariant

> No change may create a code path where a plaintext amount, spending key, or
> raw credential reaches the backend.

Proofs are generated client-side. If a PR needs the server to see a secret to
work, the design is wrong — rework the design, don't weaken the invariant.

## Prerequisites

- Rust (stable) with the `wasm32v1-none` target:
  ```bash
  rustup target add wasm32v1-none
  ```
- [Stellar CLI](https://developers.stellar.org/docs/tools/cli):
  ```bash
  cargo install --locked stellar-cli
  ```

## Building and testing

```bash
# Contracts
cd contracts
cargo test               # unit + integration tests (native)
stellar contract build   # wasm artifacts for deployment

# Circuits (arkworks toolchain)
cd circuits
cargo test
./scripts/build-artifacts.sh
```

## Testnet setup

1. Generate and fund an identity:
   ```bash
   stellar keys generate --global attesta-dev --network testnet --fund
   ```
2. Deploy in dependency order — verifiers first, then registries, then the pool:
   ```bash
   cd contracts && stellar contract build
   stellar contract deploy --wasm target/wasm32v1-none/release/attesta_zk_verifier.wasm \
     --source attesta-dev --network testnet \
     -- --circuit_id transfer --vk <VERIFYING_KEY_JSON>
   # ... issuer_registry, attestation_registry, shielded_pool
   ```
3. Verifying keys for testnet circuits are published under `circuits/artifacts/`
   (generated by `circuits/scripts/build-artifacts.sh` — M1, in progress).

## Issue taxonomy

Every issue carries:

- a **layer label**: `contract/`, `backend/`, `frontend/`, `circuits/`
- a **difficulty label**: `good-first-issue`, `help-wanted`, `core`
- **acceptance criteria** in the issue body

Anything touching circuits, verifying keys, or the no-secrets-server invariant
carries the `security-critical` label and requires **two approving reviews**.

## PR checklist

- [ ] `cargo test` passes in `contracts/` (and `circuits/` if touched)
- [ ] `cargo fmt --all` and `cargo clippy --all-targets` are clean
- [ ] `stellar contract build` succeeds
- [ ] New state transitions emit events
- [ ] Storage keys documented in the contract's module docs
- [ ] No plaintext amount / spending key / raw credential crosses the client boundary
- [ ] `security-critical` changes: soundness argument written, two reviewers requested

## Code style

- `rustfmt` defaults; `clippy` warnings are treated as errors in CI.
- Contract errors are typed (`#[contracterror]`) — no bare panics for
  user-reachable failures.
- Every admin path is evented and (pre-mainnet) will sit behind the timelocked
  multi-sig; do not add un-evented admin functions.
